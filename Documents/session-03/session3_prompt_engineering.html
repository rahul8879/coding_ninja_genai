The Root Problem — Let's Diagnose

Your prompt was: "Classify this email: {email_text}"

You gave the AI zero instructions. No categories. No format. No context. It had to guess everything:

  • What categories should it use? (It invented its own)
  • How long should the output be? (Sometimes 1 word, sometimes 200 words)
  • Should it explain its reasoning? (Sometimes yes, sometimes no)
  • What should it do with ambiguous emails? (Ask for clarification? Flag them?)
The Core Insight

The AI is not broken. Your communication is broken. This is like hiring a brilliant new analyst and telling them: "Go through these emails." Without telling them what categories to use, what format to respond in, or what to do with edge cases. They'd produce exactly this chaos.

This is the problem that Prompt Engineering solves — and we're going to fix this classifier step-by-step over the next 2 sessions.

What is
Prompt Engineering?

Before we fix our classifier, we need to understand the fundamentals. What is a prompt? Why does how you phrase things change everything?

A Prompt is a Communication Contract

A prompt is not just "a question you type to ChatGPT." It's the complete specification you give a language model — telling it who it is, what to do, what data to use, and how to format its response.

Think of it like a work ticket

If you raised a Jira ticket that just said "Do the thing with the emails", your team would laugh at you. A real ticket has: context, acceptance criteria, definition of done, and format expectations.

A good prompt is the same thing. Prompt Engineering is the skill of writing great work tickets for AI.

The Same Model, Wildly Different Results

Let's prove this with our email classifier. Same GPT-4o-mini model. Same email. Completely different prompts:

❌ Naive Prompt (What we wrote)

"Classify this email: My payment failed but I was charged twice."

✅ Engineered Prompt (What we'll build)

"You are an expert support email classifier. Classify the email into exactly one of: [Billing, Technical, Feature Request, Spam, Other]. Return only the category name, nothing else. Email: My payment failed..."

frustrated... I would suggest checking payment gateway logs..."
(127 words. Unusable in production.)
Output from Engineered Prompt
Billing
The Lesson

Prompt Engineering is the craft of structuring your instructions so the AI understands: its role, the exact task, what data to use, and precisely how to respond. Same model. Same email. Completely different quality.

The 4 Components
of Every Great Prompt

Every production-grade prompt is built from the same 4 ingredients. Miss any one, and you'll get unpredictable output. Let's fix our classifier one component at a time.

🎭
A. Role

Who should the AI be? Sets the persona, expertise level, and mental model.

📋
B. Instruction

What exact task should it perform? The core command — specific and unambiguous.

📥
C. Input

What data does it need? Context, text, or variables the AI should process.

📤
D. Output Format

Exactly how should the response look? Length, structure, tone, what to exclude.

from openai import OpenAI

client = OpenAI(api_key="your-api-key")

def classify_email(subject, sender, body):
    prompt = f"""You are an expert support email classifier 
for a SaaS product company.

Classify the email into EXACTLY ONE of these categories:
- Billing
- Technical  
- Feature Request
- Spam
- Other

Rules:
- Return ONLY the category name. Nothing else.
- No explanation, no punctuation, no extra words.
- If unsure, return "Other".

EMAIL TO CLASSIFY:
Subject: {subject}
From: {sender}
Body: {body}"""
    
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}],
        temperature=0  # deterministic output
    )
    return response.choices[0].message.content.strip()

# Test it:
result = classify_email(
    subject="Payment charged twice",
    sender="user@company.com",
    body="My payment failed but I was charged twice."
)
print(result)  # Output: "Billing" ✓
Now it works consistently!

By adding all 4 components, we went from 127-word unpredictable essays to clean, single-word outputs that your Python code can actually if result == "Billing": against. This is the power of prompt structure.

Prompt Structuring

Our classifier works — but as requirements grow, the prompt becomes a mess. How do you write prompts that stay readable, maintainable, and debuggable as they scale?

New Requirements Land on Monday

Priya is back: "Great work! Can we also detect the urgency level? And if it's a billing issue, flag whether it's about a refund or a failed payment? Oh, and sometimes emails come in Hindi — handle that too."

Your current prompt is already 12 lines. Adding all this will make it a wall of text. You need to learn to structure prompts like you structure code.

Formatting Conventions That Actually Matter

The Structured Classifier Prompt

v5_structured.pySTRUCTURED
def build_classifier_prompt(subject, sender, body):
    return f"""
# ROLE
You are an expert support email classifier for a B2B SaaS company.
You have 10 years of experience triaging customer support emails.

# TASK  
Classify the incoming email into exactly one primary category,
one urgency level, and one sub-category where applicable.

# CATEGORIES
Primary (pick ONE):
- Billing
- Technical
- Feature Request
- Spam
- Other

Urgency (pick ONE):
- High    (customer cannot use the product)
- Medium  (issue exists but workaround available)
- Low     (question or minor inconvenience)

Billing Sub-category (only if Primary = Billing):
- Refund Request
- Failed Payment
- Pricing Question
- Invoice Issue
- Not Applicable

# RULES
- RETURN ONLY valid JSON. No explanation. No markdown.
- If the email is in any language other than English, still classify it.
- If you cannot determine the category, use "Other" and urgency "Low".

# OUTPUT FORMAT
Return ONLY this JSON structure, nothing else:
{{
  "category": "[Primary Category]",
  "urgency": "[Urgency Level]",
  "billing_sub": "[Sub-category or Not Applicable]"
}}

# EMAIL TO CLASSIFY
Subject: {subject}
From: {sender}
Body: {body}
"""

# Now test with a real scenario:
prompt = build_classifier_prompt(
    subject="Urgent: Team can't login - product demo in 2 hours!",
    sender="cto@bigclient.com",
    body="Our entire team is locked out. SSO is broken. We have a board demo in 2 hours. This is critical."
)

# Expected output:
# {
#   "category": "Technical",
#   "urgency": "High",
#   "billing_sub": "Not Applicable"
# }
What Changed

The prompt now reads like a specification document. Sections are clearly labeled. Rules are explicit. Output format is machine-parseable JSON. Any developer on your team can read and modify this without breaking it — that's maintainable prompt engineering.

The Loss in Middle
Bug Nobody Talks About

Your structured prompt works great on short emails. Then someone forwards a 3-page email thread and suddenly the classifier ignores your most important rules. Here's why.

The Bug Report

Your classifier runs fine for 3 weeks. Then Rohan from the support team reports: "For long email threads, the AI is outputting full paragraphs again instead of JSON. And it's ignoring the 'RETURN ONLY JSON' rule completely."

You reproduce it immediately. Any email body longer than ~800 tokens causes the format to break. Why?

Why LLMs Forget the Middle

Language models process tokens sequentially. But their attention mechanism doesn't treat all positions equally. Research shows a U-shaped attention pattern:

attention_pattern.txt — Research finding
High Attention  ████████████░░░░░░░░░░░░░░░░░░░░░░░░░░░████████████
                  ↑                                              ↑
              Beginning                                         End
              of prompt                                      of prompt

Low Attention   ░░░░░░░░░░░░████████████████████████░░░░░░░░░░░░░░
                               ↑
                           THE MIDDLE
                     (Your critical rules
                      buried here get ignored)

The Real Bug in Our Classifier

loss_in_middle_bug.pyTHE BUG
# Our prompt structure had this problem:

prompt = """
# ROLE (Model reads this carefully — it's at the START)
You are an expert classifier...

# RULES (Buried in the middle — gets less attention)
- RETURN ONLY valid JSON.    ← This gets ignored for long emails!
- No explanation. No markdown.

# EMAIL TO CLASSIFY  
Subject: Re: Re: Re: Fwd: Re: Account Issue...
Body: [3-page email thread with 800+ tokens here]
                                                  ← END of prompt
"""
# The model reads the start carefully, skims the RULES in the middle,
# and then focuses on the long email body at the end.
# Result: JSON format ignored, free-text output returned.

4 Fixes — Applied to Our Classifier

1

Fix: Move Critical Rules to the Top

fix1_rules_first.py
prompt = f"""
CRITICAL: Return ONLY valid JSON. No explanation. No markdown fences.

# ROLE
You are an expert support email classifier...

[rest of prompt]
"""
# Critical constraint is now the FIRST thing the model sees.
2

Fix: Repeat Key Rule at the End (Recency Bias)

fix2_repeat_at_end.py
prompt = f"""
CRITICAL: Return ONLY valid JSON. No explanation.

[...rest of prompt...]

# EMAIL TO CLASSIFY
Body: {email_body}

REMINDER: Return ONLY the JSON object below. Nothing before it. Nothing after.
{{
  "category": "...",
  "urgency": "...",
  "billing_sub": "..."
}}
"""
# Rule appears at START (primacy) AND END (recency) — model can't miss it.
3

Fix: Limit Email Length Before Sending to API

fix3_truncate.py
def preprocess_email(body, max_chars=1500):
    """Truncate long email threads — classifier only needs key context."""
    if len(body) > max_chars:
        # Keep the first 1000 chars (most recent message)
        # and last 500 chars (usually has context/subject)
        body = body[:1000] + "\n...[truncated]...\n" + body[-500:]
    return body

# Now the "middle" is much shorter — less risk of loss.
4

Fix: Use System Role for Critical Rules (OpenAI API)

fix4_system_role.py
response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[
        {
            "role": "system",  # System prompt = always attended to
            "content": """You are an expert support email classifier.
ALWAYS return valid JSON only. Never return free text."""
        },
        {
            "role": "user",
            "content": build_classifier_prompt(subject, sender, body)
        }
    ],
    temperature=0
)
Why System Role Works

The system role in the OpenAI API has special treatment — models are trained to pay extra attention to system instructions. Putting your non-negotiable constraints there means they rarely get ignored, even with long user messages.

Prompt Crafting

Structure tells the AI what to do. Crafting tells it how to think while doing it. The difference between "works" and "works brilliantly" is in the wording.

New Problem — Edge Case Hell

Your classifier handles clean emails perfectly. But real-world emails are messy: "Hi, the app is slow AND I want to request a refund — which is it?". Or passive-aggressive ones: "Wow, yet another billing failure. Fantastic product." (Sarcasm. Billing issue. Your AI classifies it as "Feature Request.")

Structure alone can't fix this. You need to craft the model's reasoning process.

The 5 Crafting Levers

1

Clarity & Conciseness — Remove ambiguity ruthlessly

❌ Ambiguous

"If the email seems like it could be billing related, classify it appropriately."

✅ Clear

"If the email mentions payment, invoice, charge, refund, or subscription cost → classify as Billing."

2

Specificity — Name every edge case you know about

crafting_specificity.py
# Before: Generic
"Classify the email into the correct category."

# After: Specific with edge cases named
"""Classify the email using these rules:
- If the email mentions BOTH a billing issue and a technical issue, 
  classify by whichever the customer explicitly complains about first.
- Sarcastic complaints about billing (e.g. "great job charging me twice")
  should still be classified as Billing, not Other.
- Auto-reply emails (e.g. "Out of office: I am away until...") → Other
- Emails from @noreply or @automated → Spam"""
3

Tone & Behavior — Tell it how to "think"

❌ No Behavior Guidance

Model might be overly conservative and put everything in "Other" when unsure.

✅ Explicit Behavior

"When in doubt between two categories, prefer the one with higher business impact (Billing > Technical > Feature Request > Other)."

4

Depth Control — Prevent over-explaining

crafting_depth.py
# Constraining output prevents the "essay mode" problem
"""DO NOT include:
- Explanations of why you chose the category
- Suggestions for how the support team should respond  
- Any text outside the JSON structure
- Markdown code blocks or backticks

ONLY return the JSON object. The response should be 3-5 lines maximum."""
5

Intent Framing — The "Act like a senior analyst" technique

crafting_intent.py
"""You are an expert support email classifier with 10 years of experience 
at a SaaS company. You have seen every type of customer email.

Before classifying, mentally ask yourself:
1. What is the customer's PRIMARY frustration?
2. What action does this email require from our team?
3. Does this match a known pattern (billing complaint, crash report, etc.)?

Then classify based on these answers, not just keywords."""

# Intent framing makes the model reason from the customer's perspective
# instead of just keyword-matching. Handles sarcasm and ambiguity much better.
Why This Works

"Wow, yet another billing failure" — with intent framing, the model now asks: "What is the primary frustration? Billing. What action does this require? Billing team intervention." Category: Billing ✓. Without it, the model sees "fantastic product" and might misclassify.

Zero-Shot, Few-Shot
& Role-Based
Techniques

Structure and crafting got our classifier to ~70% accuracy. The next 15% improvement comes from teaching the model using examples — and from the right role assignment. Let's see the real impact in code.

Zero-Shot — What We've Been Doing

Our current classifier is zero-shot — we describe the task but give no examples. The model relies entirely on its training data.

zero_shot_classifier.py
# Zero-Shot: Task description only, no examples
prompt = """
You are a support email classifier. 
Classify emails into: Billing, Technical, Feature Request, Spam, Other.
Return JSON only.

Email: {email}
"""
# Works for straightforward emails.
# Struggles with:
# - Industry-specific jargon ("our SFDC integration is broken")
# - Your company-specific categories 
# - Ambiguous cases unique to your domain

Few-Shot — Teach By Example

Your classifier was mishandling "SFDC integration broken" (Salesforce = CRM = partner integration issue, not just "Technical"). Your general training data doesn't know your business context. Examples do.

few_shot_classifier.pyIMPROVED
def build_few_shot_prompt(email_subject, email_body):
    # These are REAL examples from your actual ticket history
    examples = [
        {
            "subject": "Charged $299 instead of $99",
            "body": "I upgraded to the Pro plan but you charged me $299, the Enterprise price.",
            "output": {"category": "Billing", "urgency": "High", "billing_sub": "Failed Payment"}
        },
        {
            "subject": "SFDC integration not syncing",
            "body": "Our Salesforce integration stopped syncing leads 3 days ago. Last sync: Nov 12.",
            "output": {"category": "Technical", "urgency": "High", "billing_sub": "Not Applicable"}
        },
        {
            "subject": "Can you add Slack notifications?",
            "body": "Would love to get a Slack ping when a new lead comes in. Is this planned?",
            "output": {"category": "Feature Request", "urgency": "Low", "billing_sub": "Not Applicable"}
        },
        {
            "subject": "WIN $500 AMAZON GIFT CARD!!!",
            "body": "Click here to claim your prize NOW limited time offer!!!!",
            "output": {"category": "Spam", "urgency": "Low", "billing_sub": "Not Applicable"}
        },
    ]
    
    # Build the examples block
    examples_block = ""
    for i, ex in enumerate(examples, 1):
        examples_block += f"""
EXAMPLE {i}:
Subject: {ex['subject']}
Body: {ex['body']}
Output: {ex['output']}
"""
    
    return f"""
CRITICAL: Return ONLY valid JSON. No explanation. No markdown.

# ROLE
You are a support email classifier for a B2B SaaS company with deep 
knowledge of our product and customer patterns.

# EXAMPLES OF CORRECT CLASSIFICATIONS
Study these examples carefully — they reflect OUR specific business context:
{examples_block}

# NOW CLASSIFY THIS EMAIL
Subject: {email_subject}
Body: {email_body}

REMINDER: Return ONLY the JSON. Exactly 3 fields: category, urgency, billing_sub.
"""

# Test with the edge case that was failing before:
result = classify_email(
    build_few_shot_prompt(
        "Our SFDC won't sync",
        "Salesforce stopped pulling contacts since Tuesday."
    )
)
# Now correctly returns: Technical / High / Not Applicable
Why Few-Shot Beats Zero-Shot Here

Your training data knows what "Salesforce" is, but it doesn't know that your company treats SFDC integration issues as High-urgency Technical tickets. The examples encode your business-specific knowledge into the prompt — no model fine-tuning needed.

Role-Based — Beyond Just "You are a classifier"

New Requirement: Priority Scoring

Priya: "Can the AI also score tickets 1-10 for priority? Our high-value customers (paying $1000+/month) should get their tickets bumped up."

This requires a different kind of expertise — customer success, not just classification. Same classifier, different role.

In production systems, you'll often chain multiple role-based prompts: Classifier → Priority Scorer → Response Drafter → Quality Checker. Each role is an expert at one thing. This is how real agentic systems work — and we'll build exactly this in Sessions 4 and 5.

Dynamic
Prompting

Your team loves the classifier. Now the CEO asks: "Can we deploy this for our 3 different products? Each has different categories and customer types." Static prompts break here. Dynamic prompts scale.

The Scaling Problem

You currently have one hardcoded prompt for one product. The CEO just acquired two companies — now you need classifiers for a fintech product, an HR tool, and your original SaaS. Each has different categories, urgency rules, and customer SLAs. You can't maintain 3 separate prompts manually.

Static vs Dynamic Prompts

❌ Static (What we have)

Prompt hardcoded in Python file. Change a category → find and edit in 3 places. Add a new product → copy-paste the whole thing. Team makes typos. Bugs appear.

✅ Dynamic (What we'll build)

Prompt assembled at runtime from config. Adding a product = add a JSON config file. Changing categories = one config change, all prompts update automatically.

Dynamic Prompt Factory — Production Pattern

dynamic_classifier.pySCALABLE
from openai import OpenAI
import json

client = OpenAI(api_key="your-api-key")

# Product configs — each product team owns their config
PRODUCT_CONFIGS = {
    "saas_crm": {
        "product_name": "CRM Platform",
        "categories": ["Billing", "Technical", "Feature Request", "Spam", "Other"],
        "urgency_levels": ["High", "Medium", "Low"],
        "sla_hours": {"High": 2, "Medium": 8, "Low": 24},
        "examples": [
            {"subject": "SFDC not syncing", "body": "Salesforce stopped pulling leads",
             "output": {"category": "Technical", "urgency": "High"}}
        ]
    },
    "fintech_payments": {
        "product_name": "Payment Gateway",
        "categories": ["Transaction Failed", "Fraud Alert", "Settlement Delay", "KYC Issue", "Other"],
        "urgency_levels": ["Critical", "High", "Medium"],
        "sla_hours": {"Critical": 1, "High": 4, "Medium": 12},
        "examples": [
            {"subject": "Payment declined at checkout", "body": "Card declined even though sufficient balance",
             "output": {"category": "Transaction Failed", "urgency": "High"}}
        ]
    }
}

def build_dynamic_prompt(product_key, subject, body):
    """Build a product-specific prompt from config. Zero hardcoding."""
    config = PRODUCT_CONFIGS[product_key]
    
    # Dynamically build categories list
    categories_str = "\n".join([f"  - {cat}" for cat in config["categories"]])
    
    # Dynamically build examples block
    examples_str = ""
    for i, ex in enumerate(config["examples"], 1):
        examples_str += f"\nEXAMPLE {i}:\nSubject: {ex['subject']}\nBody: {ex['body']}\nOutput: {json.dumps(ex['output'])}\n"
    
    # Build SLA context
    sla_str = ", ".join([f"{k}: {v}hr response" for k, v in config["sla_hours"].items()])
    
    return f"""CRITICAL: Return ONLY valid JSON.

# ROLE
You are an expert support email classifier for {config['product_name']}.

# CATEGORIES
Classify into exactly one of:
{categories_str}

# URGENCY LEVELS  
Use exactly one of: {', '.join(config['urgency_levels'])}
SLAs: {sla_str}

# EXAMPLES FROM THIS PRODUCT'S HISTORY
{examples_str}

# EMAIL TO CLASSIFY
Subject: {subject}
Body: {body}

REMINDER: Return ONLY JSON with keys "category" and "urgency"."""

# Using it — one function, any product:
crm_result = build_dynamic_prompt("saas_crm", "Login broken", "Can't sign in since morning")
fin_result = build_dynamic_prompt("fintech_payments", "Payment failed", "Card declined at ₹999 checkout")

# Adding a 3rd product = just add a new key to PRODUCT_CONFIGS.
# The prompt function never changes. That's the power of dynamic prompting.
What "Dynamic" Really Means

Dynamic prompting is the bridge between prompt engineering and software engineering. When prompts are data-driven, they're: versionable (git commit the configs), testable (unit test each product's prompts), scalable (10 products = 10 config files), and maintainable (product teams own their configs without touching Python).

This is exactly how LangChain's PromptTemplate works — which we'll cover in Session 5.
