The Intern Test

Thought Experiment

You have two interns. You give both the same email: "I can't login and I already paid for annual plan last week."

Intern A (pattern matcher) reads it and says: "Login problem → Technical." Done. 3 seconds.

Intern B (reasoner) thinks out loud: "They paid last week. This is a recent payment. If there's a payment processing delay, the account might not have been activated yet. So this is technically a login failure, but the root cause is billing. The billing team would need to manually activate the account. I'll classify as Billing."

Who do you trust with real tickets? Intern B every time. Chain of Thought teaches the AI to be Intern B.

Chain of Thought
Prompting

We add one thing to our classifier: ask it to reason out loud before answering. This single change fixes Failure Types 1, 2, and 3. Here's exactly how to implement it.

The Core Idea

Chain of Thought (CoT) prompting asks the model to produce its intermediate reasoning steps before giving the final answer. Instead of email → category, it becomes email → reasoning → category. The model that shows its work almost always gives better final answers.

What Happens Inside the Model

LLMs predict the next token. When you add "Think through this step by step before classifying", the model has learned during training that this phrase is followed by structured reasoning. It shifts from pattern completion mode to reasoning mode — and the next tokens it predicts are reasoning steps rather than a quick answer.

❌ Without CoT (Pattern Mode)

Model sees "can't login" → immediately predicts "Technical" as the most statistically likely next token. Fast, often wrong.

✅ With CoT (Reasoning Mode)

Model predicts a reasoning chain first: "They mention a recent payment... account activation... root cause = Billing..." Then predicts "Billing" as the answer.

Fixing Failure Type 1 with Zero-Shot CoT

The simplest form: add "Think step by step" — no examples needed.

Auto-CoT — When You Have Too Many Categories

Your fintech client now has 12 email categories. Writing Few-Shot CoT examples for all 12 combinations takes hours. Auto-CoT makes the model generate its own reasoning examples first, then classify.

auto_cot_classifier.pyAUTO-COT
def auto_cot_classify(email_subject, email_body):
    
    ### PHASE 1: Model generates its own reasoning warm-up ###
    warmup_prompt = """You are a fintech support analyst. 
    
Before I give you a real email to classify, I want you to demonstrate 
your reasoning ability. Generate 3 different example emails a fintech 
support team might receive, and show step-by-step how you would 
classify each one. 

Cover different categories: payment failures, KYC issues, and fraud alerts.

Format each as:
EXAMPLE [N]:
Email: [email text you invent]
Reasoning: [step-by-step thinking]
Classification: [category + urgency]"""
    
    # Get the model's self-generated examples
    warmup_response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": warmup_prompt}],
        temperature=0.3  # slight randomness for diverse examples
    )
    self_examples = warmup_response.choices[0].message.content
    
    ### PHASE 2: Use those examples to classify real email ###
    classify_prompt = f"""You are a fintech support analyst.

You just demonstrated your reasoning ability with these examples:
{self_examples}

Now apply the SAME reasoning pattern to this real email:

Subject: {email_subject}
Body: {email_body}

Follow the same format: Reasoning → Classification JSON."""
    
    classify_response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": classify_prompt}],
        temperature=0
    )
    return classify_response.choices[0].message.content

# Auto-CoT = model teaches itself → model applies to real case
# Eliminates hours of manual example writing
# Trade-off: slightly more expensive (2 API calls per classification)
💡 When to Use Which CoT Type
  • Zero-Shot CoT: General emails where the model has enough domain knowledge. "Think step by step" is usually enough.
  • Few-Shot CoT: Domain-specific emails with terminology the model hasn't learned (fintech, healthcare, legal). Give it 3-5 reasoning examples.
  • Auto-CoT: When you have many categories and can't manually write examples for all. Let the model warm up first. Costs 2 API calls.

Self-Consistency

CoT improved accuracy significantly. But you discover a new problem: on ambiguous emails, the model's answer changes between runs. Same email, different classification. You need reliability, not just reasoning.

The Reliability Bug

You add monitoring to your classifier. Over 3 days, you notice: for 8% of emails, running the same prompt twice gives different answers. One ambiguous email gets classified "Billing" on Monday, "Technical" on Tuesday. The support routing is inconsistent. An Enterprise client notices.

Your CoT reasoning is good — but LLMs are probabilistic. Even at temperature=0, slightly different email phrasings or model version updates can cause drift. You need a way to get stable answers on ambiguous cases.

The Solution

Self-Consistency: Run the same prompt multiple times (with slight randomness), collect all answers, and pick the one that appears most frequently. If the model is right, it will be consistently right. If it's uncertain, you can detect and handle that uncertainty.

Implementing Self-Consistency for the Classifier

self_consistency_classifier.pySELF-CONSISTENCY
from collections import Counter
import json, re

def classify_with_self_consistency(subject, body, n_runs=5):
    """
    Run classifier N times, return majority answer.
    Also returns confidence score and whether human review is needed.
    """
    results = []
    raw_outputs = []
    
    for run in range(n_runs):
        # Use temperature > 0 to get variation in reasoning paths
        # Different paths, same right answer = high confidence
        response = client.chat.completions.create(
            model="gpt-4o-mini",
            messages=[{
                "role": "user",
                "content": build_cot_prompt(subject, body)
            }],
            temperature=0.3  # slight randomness = different reasoning paths
        )
        
        raw = response.choices[0].message.content
        raw_outputs.append(raw)
        
        # Extract the JSON from CoT output
        json_match = re.search(r'\{[^}]+\}', raw)
        if json_match:
            try:
                parsed = json.loads(json_match.group())
                results.append(parsed.get("category", "Unknown"))
            except:
                results.append("ParseError")
    
    # Count votes
    vote_counts = Counter(results)
    winner = vote_counts.most_common(1)[0]
    winner_category, winner_votes = winner
    
    confidence = winner_votes / n_runs  # e.g. 4/5 = 80% confidence
    needs_review = confidence < 0.6     # flag if no clear majority
    
    return {
        "category": winner_category,
        "confidence": f"{confidence:.0%}",
        "vote_breakdown": dict(vote_counts),
        "needs_human_review": needs_review,
        "reasoning_traces": raw_outputs  # audit trail!
    }

# Test on the churn-risk email:
result = classify_with_self_consistency(
    subject="Evaluating if we stay on your platform",
    body="Zapier missing, bulk export missing, prices jumped 40%.",
    n_runs=5
)
print(result)

def smart_classify(email):
    # Step 1: Try single CoT run first

Tree of
Thought

Self-Consistency handles ambiguous emails by running the same reasoning 5 times. But some emails need a fundamentally different approach — they need to explore multiple interpretations simultaneously and pick the best one.

A New Use Case: Response Drafting

Priya: "Now that we classify emails reliably, can we also auto-draft responses for the support team? They still need to write each reply manually."

This is a different problem. Classification has one right answer. Drafting a response has many valid approaches — empathetic vs. direct, detailed vs. concise, offering refund vs. asking for more info. Self-Consistency can't help — it votes on one right answer. You need to explore multiple paths, evaluate each, and pick the best. That's Tree of Thought.

Tree of Thought Defined

Tree of Thought (ToT) prompts the model to generate multiple reasoning branches, evaluate each branch for quality/suitability, and then select (or combine) the best path. Unlike CoT's linear reasoning, ToT explores the solution space before committing.

ToT for Response Drafting

def process_email(email: dict) -> dict:
    """
    Complete email intelligence pipeline.
    Combines all techniques learned across Sessions 3 & 4.
    
    Returns:
    - classification (category + urgency + confidence)
    - routing decision  
    - drafted response (for churn-risk emails)
    - audit trail (reasoning traces for debugging)
    """
    
    ### STEP 1: Quick classification with CoT (1 API call) ###
    initial = classify_with_cot(
        subject=email["subject"],
        body=email["body"]
    )
    
    ### STEP 2: Check if this is a high-stakes case ###
    is_high_stakes = (
        initial["urgency"] == "High"
        or is_enterprise_customer(email["sender"])
        or any(w in email["body"].lower()
               for w in ["cancel", "competitor", "lawsuit", "refund"])
    )
    
    ### STEP 3: If high-stakes, run Self-Consistency for reliability ###
    if is_high_stakes:
        final_classification = classify_with_self_consistency(
            email, n_runs=5
        )
    else:
        final_classification = initial  # CoT is enough
    
    ### STEP 4: Route the ticket ###
    routing = {
        "Billing": "billing-team@company.com",
        "Technical": "tech-support@company.com",
        "Feature Request": "product@company.com",
        "Spam": None,  # auto-delete
    }
    routed_to = routing.get(final_classification["category"], "support@company.com")
    
    ### STEP 5: If churn-risk, draft response using Tree of Thought ###
    drafted_response = None
    is_churn_risk = (
        "cancel" in email["body"].lower()
        or "competitor" in email["body"].lower()
    )
    
    if is_churn_risk:
        drafted_response = draft_response_with_tot(email, final_classification)
    
    ### RETURN everything ###
    return {
        "email_id": email["id"],
        "classification": final_classification,
        "routed_to": routed_to,
        "drafted_response": drafted_response,
        "needs_human_review": final_classification.get("needs_human_review", False),
        "audit_trail": final_classification.get("reasoning_traces", [])
    }
